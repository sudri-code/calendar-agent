# PRD — Telegram-бот для управления Exchange-календарями

## 1. Общее описание

### 1.1 Название продукта

Telegram Calendar Sync Bot

### 1.2 Цель

Создать Telegram-бота на Python, который помогает пользователю управлять несколькими календарями Exchange / Outlook через Telegram:

- создавать встречи в выбранном календаре;
- автоматически зеркалировать занятый слот в других подключённых календарях;
- учитывать занятость участников встречи;
- разбирать текстовые команды на русском языке через LLM;
- предоставлять быстрый UI через кнопки и inline-календарь.

### 1.3 Проблема

У пользователя несколько рабочих календарей. Если встреча назначается в одном календаре, слот должен быть синхронно заблокирован и в остальных, чтобы не возникало пересечений.  
Дополнительно пользователь хочет создавать встречи из свободного текста, например:  
`Создай встречу с Ваней Ивановым на этой неделе`.

### 1.4 Целевая аудитория

- специалисты и руководители с несколькими рабочими Exchange-календарями;
- пользователи, которым нужен быстрый интерфейс из Telegram без постоянного открытия Outlook.

---

## 2. Product Vision

Бот должен стать единым интерфейсом управления несколькими Exchange-календарями:

- понимает естественный русский язык;
- проверяет доступность всех нужных календарей и участников;
- предлагает альтернативные слоты;
- сохраняет консистентность между календарями при создании, переносе и удалении событий.

---

## 3. Scope

## 3.1 MVP

В MVP входят:

1. Авторизация пользователя в нескольких Exchange-аккаунтах через Microsoft OAuth.
2. Подключение произвольного числа календарей.
3. Получение списка календарей и контактов из всех подключённых аккаунтов.
4. Объединение дублей контактов.
5. Создание события через Telegram:
   - выбор календаря;
   - ввод параметров вручную или текстом;
   - проверка свободности:
     - выбранного календаря;
     - остальных подключённых календарей пользователя;
     - участников встречи;
   - создание основного события;
   - создание зеркальных событий-блокировок в остальных календарях.
6. Обновление события:
   - перенос времени;
   - изменение длительности;
   - изменение состава участников;
   - повторная проверка доступности;
   - синхронное обновление зеркальных событий.
7. Удаление события:
   - удаление основного события;
   - удаление всех зеркальных событий.
8. Telegram UI:
   - команды;
   - reply / inline кнопки;
   - inline-календарь для выбора даты;
   - быстрые сценарии создания встречи.
9. Поддержка LLM через OpenRouter для разбора текстовых запросов на русском языке.
10. Логи и журнал ошибок.

## 3.2 Out of scope для MVP

Не входят в MVP:

- веб-панель администратора;
- Google Calendar;
- создание Teams/Zoom/Meet ссылок;
- групповые корпоративные календари как отдельный объект синхронизации;
- сложные правила приоритизации календарей;
- голосовой ввод;
- аналитика загрузки по неделям/месяцам.

---

## 4. Основные пользовательские сценарии

### 4.1 Создание встречи текстом

Пользователь пишет:
`Создай встречу с Ваней Ивановым на этой неделе`

Бот:

1. отправляет запрос в LLM;
2. извлекает намерение и неполные параметры;
3. уточняет недостающие данные:
   - день;
   - время;
   - длительность;
   - календарь;
4. находит контакт(ы);
5. проверяет занятость:
   - всех подключённых календарей пользователя;
   - участников;
6. предлагает подходящие слоты;
7. после подтверждения создаёт событие в выбранном календаре;
8. создаёт блокировки в остальных календарях;
9. возвращает итог пользователю.

### 4.2 Создание встречи кнопками

Пользователь нажимает:
`Создать встречу`

Бот последовательно предлагает:

- календарь;
- дату;
- время;
- длительность;
- участников;
- название;
- описание.

### 4.3 Перенос встречи

Пользователь выбирает встречу и меняет время.  
Бот повторно проверяет свободность всех календарей и участников, затем переносит основное событие и все зеркальные.

### 4.4 Удаление встречи

Пользователь удаляет встречу.  
Бот удаляет основное событие и все связанные зеркальные события.

### 4.5 Управление учётными записями

Пользователь открывает настройки и может:

- подключить Exchange-аккаунт;
- отключить аккаунт;
- выбрать календари для синхронизации;
- обновить авторизацию;
- запустить повторную синхронизацию контактов.

## 5. Functional Requirements

## 5.1 Управление аккаунтами

Система должна:

- поддерживать произвольное число Exchange-аккаунтов на одного пользователя;
- хранить токены доступа и обновления в зашифрованном виде;
- уметь обновлять access token через refresh token;
- отмечать статус подключения аккаунта;
- поддерживать переподключение аккаунта при истечении прав.

## 5.2 Управление календарями

Система должна:

- получать список календарей из каждого аккаунта;
- позволять пользователю выбрать, какие календари участвуют в синхронизации;
- хранить флаг:
  - основной календарь по умолчанию;
  - активен ли календарь для зеркалирования.

## 5.3 Управление контактами

Система должна:

- получать контакты из всех подключённых аккаунтов;
- объединять дубли по email и нормализованному имени;
- позволять искать контакты по имени, email, части имени;
- показывать источник контакта;
- учитывать возможность отсутствия контакта и позволять ввести email вручную.

Microsoft Graph поддерживает получение контактов из папки контактов пользователя, а для календарей и событий — операции чтения/записи по Outlook Calendar API. :contentReference[oaicite:1]{index=1}

## 5.4 Создание событий

При создании события система должна:

1. получить или собрать параметры:
   - title;
   - date;
   - start time;
   - duration;
   - attendees;
   - description;
   - target calendar;
2. проверить доступность:
   - выбранного календаря;
   - остальных активных календарей пользователя;
   - участников встречи;
3. если слот недоступен — предложить альтернативы;
4. после подтверждения:
   - создать основное событие в выбранном календаре;
   - создать зеркальные события в остальных календарях.

Для создания событий Microsoft Graph предоставляет endpoint создания события; для проверки занятости пользователей и ресурсов используется getSchedule / related scheduling capabilities. :contentReference[oaicite:2]{index=2}

## 5.5 Правила зеркалирования

Зеркальное событие:

- создаётся во всех прочих активных календарях пользователя;
- не содержит участников;
- должно содержать в описании сведения об исходной встрече:
  - название исходной встречи;
  - время;
  - список участников;
  - идентификатор связки;
  - указание, в каком календаре находится основное событие.

Пример описания зеркального события:
`Зеркальная блокировка. Основная встреча создана в календаре "Work A". Участники: ivan@example.com, petr@example.com. Sync group: abc123.`

## 5.6 Синхронизация изменений

При изменении основного события система должна:

- получить уведомление или обнаружить изменение;
- заново проверить доступность слота в других календарях и у участников;
- обновить зеркальные события;
- если новый слот конфликтует — зафиксировать конфликт и уведомить пользователя с альтернативами.

При удалении основного события система должна:

- удалить зеркальные события;
- очистить mapping.

При изменении зеркального события вручную в Outlook:

- система должна считать это нежелательным сценарием;
- восстановить консистентность по правилам sync engine;
- уведомить пользователя о расхождении.

## 5.7 Проверка занятости участников

Система должна уметь:

- определять участников по контактам;
- запрашивать свободность участников через Microsoft Graph scheduling API;
- учитывать занятость самого пользователя по всем активным календарям;
- предлагать N лучших альтернативных слотов.

Важно: подбор слотов должен опираться не только на «мой календарь», но и на занятость приглашённых участников. Microsoft Graph предоставляет механизмы работы с free/busy и поиском доступных временных окон. :contentReference[oaicite:3]{index=3}

## 5.8 Обработка текста через LLM

Система должна поддерживать русский язык и разбирать запросы вида:

- `Поставь встречу завтра в 15:00 на 30 минут с Иваном`
- `Создай встречу с Ваней Ивановым на этой неделе`
- `Перенеси встречу с Петром на пятницу после обеда`
- `Покажи мой день`
- `Найди окно на этой неделе на час с Олей`

LLM должен:

- определить intent;
- извлечь сущности;
- вернуть структурированный JSON;
- работать в пределах краткого контекста одной задачи.

OpenRouter поддерживает chat completions через единый API и Bearer authentication. :contentReference[oaicite:4]{index=4}

## 5.9 Telegram UX

Бот должен поддерживать:

- `/start`
- `/help`
- `/today`
- `/week`
- `/create`
- `/find_slot`
- `/reschedule`
- `/delete`
- `/settings`
- `/accounts`

Кнопки быстрого доступа:

- Создать встречу
- Найти слот
- Мой день
- Моя неделя
- Перенести
- Удалить
- Аккаунты
- Календари
- Контакты
- Настройки

Дополнительно:

- inline-календарь выбора даты;
- списки длительностей: 15 / 30 / 45 / 60 / 90 мин;
- кнопки подтверждения альтернативных слотов.

## 6. Non-Functional Requirements

### 6.1 Производительность

- обычный ответ Telegram UI: до 2 секунд без учёта ожидания внешних API;
- сценарий подбора слота: до 8 секунд в типовом случае;
- создание встречи: до 10 секунд в типовом случае.

### 6.2 Надёжность

- идемпотентность операций создания/обновления;
- защита от дублей при повторной доставке webhook;
- восстановление после временного сбоя Graph API;
- периодическая фоновая сверка связок событий.

### 6.3 Безопасность

- хранение секретов только в защищённом vault / env;
- шифрование refresh tokens и других чувствительных данных;
- ограничение Telegram-доступа только авторизованным пользователям;
- audit logging по операциям создания / обновления / удаления;
- webhook endpoint только по HTTPS.

### 6.4 Масштабируемость

На MVP достаточно вертикального масштабирования на VPS.  
Архитектура должна позволять вынести:

- bot;
- API;
- sync worker;
- DB;
- cache / queue
  в отдельные контейнеры.

---

## 7. Критерии успеха

1. Не менее 95% операций создания встречи завершаются без ручного исправления.
2. Не менее 90% текстовых запросов корректно распознаются без повторного ввода.
3. Не более 1% случаев рассинхронизации между основным и зеркальными событиями.
4. Пользователь может подключить новый Exchange-аккаунт без участия разработчика.
5. Любое удаление / перенос основной встречи корректно отражается в зеркальных календарях.

---

## 8. Ограничения и риски

1. Microsoft Graph webhook subscriptions имеют срок жизни и требуют продления/реавторизации. :contentReference[oaicite:5]{index=5}
2. Free/busy и поиск окна зависят от доступности данных по участникам и прав доступа. :contentReference[oaicite:6]{index=6}
3. Контакты из разных аккаунтов могут конфликтовать по имени и email.
4. LLM может неверно трактовать относительные даты вроде «на этой неделе вечером».
5. Возможны race conditions, если пользователь параллельно меняет встречи в Outlook и через Telegram.
